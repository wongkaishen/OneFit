1.      System Overview

1.1. Description

OneFit is a digital wellness platform designed to combat sedentary behavior, poor diet, and chronic
stress among Gen Z and Gen Alpha by integrating self-care and mental resilience into their digital-
native lifestyles. The system facilitates seamless interactions between three primary actors, which
is Gym Users, Wellness Specialists, and Administrators. So the web application is:

Gym User Functions
    - Allow users to register safe
    - Manage their fitness profiles
    - Create personalized workout plans
    - Log daily activities and dietary intake, which is daily calories
    - Update and share their progress
    - When a Wellness Specialist submits feedback, the Gym User's workout/meal plan is
         automatically updated and the user receives a notification

Wellness Specialist Functions
    - Review user progress reports and monitor overall health trends
    - Assign customized wellness tasks
    - Provide professional feedback
    - Manage educational content heavily focused on mental resilience and self-care
    - Monitor the gym user wellness community groups

Administrator Functions
    - Viewing and managing all users
    - Approving member registrations
    - Reinstating suspended memberships
    - Monitoring overall user activity
    - Removing inactive programs
    - Sending updates and announcements to members

1.3. Assumptions and Dependencies

1.3.1 Assumptions

Device Access
    - All roles (Gym Users, Wellness Specialists, and Admins) will access the web
         application using a smartphone, a tablet, or a modern computer with an active
         internet connection.

User Input Accuracy
    - The effectiveness of the AI-generated personalized workout plans, calorie tracking,
         and suitable meal recommendations assumes that the user inputs their daily
         activity, anthropometric measurements, and dietary data honestly and accurately.

Content Validity
    - The wellness specialists have the legal permissions and copyrights to upload and
         manage the educational content and mental resilience courses on the platform.

System Load
    - The system architecture will comfortably handle a maximum concurrent user base
         of around 500 users without experiencing performance degradation or UI lag.

Health Disclaimer
    - The system serves strictly as a digital wellness guide. Users must consult a certified
         medical professional before starting any new diet or intensive physical training plan
         generated by the AI.

1.3.2 Dependencies

Nutritional Database/API:

    - The system relies on a third-party nutritional API (e.g., Edamam or FatSecret) to
         fetch accurate macro and micro-nutrient data for the user's daily calorie and meal
         logging.

Database & Authentication:

    - The project relies on Supabase (PostgreSQL) for secure role-based access control
         (OAuth and Email/Password), secure user data storage, and cloud storage for user
         progress photos and educational content.

AI Inference Engine:

         the availability and uptime of a Small Language Model (SLM) accessed via a cloud

Backend Middleware:
    - The system depends on a Python/FastAPI backend to securely route sensitive
         health data between the Next.js frontend, the database, and the AI models.

2.2 Wellness Specialist

2.2.1 Use Case 1: Review User Progress Reports
The Wellness Specialist opens the Progress Review dashboard and selects a Gym User
(or a community group). The system queries Supabase for the user's weight, body-fat %,
activity and dietary logs, and milestones, then renders a progress report with trend charts.
The specialist may iteratively filter by date range; after reviewing the at-risk indicators they
either flag the user for follow-up (assign a task / provide feedback) or take no action, then
close the report.

date. Input is validated in a loop until valid, after which the system stores the task with
status Assigned, notifies the user(s), and confirms success.

2.2.4 Use Case 4: Provide Professional Feedback and Consultation

and recalculation is queued. The Gym User is then notified (this realises the system rule
that submitted feedback auto-updates the plan and notifies the user).

3.                         Data Design

3.1. Design Class Diagram

The Design Class Diagram models the static structure of the OneFit system. It shows the main
classes -- User, GymUser, WellnessSpecialist, Admin, FitnessProfile, WorkoutPlan,
WorkoutSession, ActivityLog, DietaryLog, ProgressEntry, Milestone, WellnessTask,
EducationalContent, Feedback, CommunityGroup, CommunityPost, Notification, Announcement,
HealthTrendReport, and AuditLog -- together with their attributes, methods, and relationships. It
captures generalisation (GymUser, WellnessSpecialist, and Admin all inherit from User) and
association (e.g. a GymUser owns one FitnessProfile, has many WorkoutPlans, and receives many
Notifications). This diagram is the blueprint from which the database schema and service classes
are derived.

5.                         Architecture Design

5.1. Software Architecture

5.1 Software Architecture

                                                       Page 90

    if
Software Design Speci cation for OneFit (Version 2.0)

OneFit uses a 4-tier, cloud-based architecture that cleanly separates presentation,
application, data, and external concerns:

    À Client Tier (User Device): the Next.js client runs in the browser / mobile web for all
         three roles and communicates over HTTPS.

    À Presentation Tier (Vercel): the Next.js application handles SSR, routing and UI,
         calling the backend over HTTPS/REST and using the Supabase JS SDK directly for
         auth and realtime.

    À Application Tier (FastAPI backend): the core of the system. It exposes REST
         endpoints and holds business logic, organised into service modules -- Auth &
         Identity, Gym User Service, Wellness Specialist Service, Admin Service, AI
         Orchestration, and Notification Service. Keeping logic and API credentials here is
         what protects sensitive health data.

    À Data Tier (Supabase): PostgreSQL with Row-Level Security, Supabase Auth
         (JWT), Storage buckets (progress photos, educational media) and Realtime/Edge
         for live notifications.

    À External Services: USDA Food Data (nutrition lookup), Groq Cloud (LLM plan/
         feedback generation), Hugging Face (pose/model inference), and an Email service
         (SMTP).

This layering improves scalability, maintainability and security: each tier can scale
independently and credentials for the AI and email services never leave the backend.

Separation into subsystems & assignment to team members
The system is divided by actor-domain into three feature subsystems plus two cross-
cutting subsystems.

5.1.1 AI & Integration Subsystem

This subsystem orchestrates the system's third-party AI and data services. It routes plan-
and feedback-generation requests to Groq Cloud, pose/model inference to Hugging Face,
and nutrition lookups to USDA FoodData Central, validating and normalising each
response before returning it to the calling service. Centralising these integrations keeps
external API credentials on the backend and isolates the rest of the system from provider-
specific details.

5.1.2 Gym User Subsystem

This subsystem handles the end-user experience. It manages registration and login, profile
and anthropometric data, daily activity and dietary logging, AI-generated workout/meal
plans, and progress tracking and sharing. It is the primary consumer of the outputs
produced by the Wellness Specialist and AI subsystems, such as assigned tasks,
feedback, updated plans and notifications.

5.1.3 Wellness Specialist Subsystem

This subsystem supports the Wellness Specialist (health coach / nutritionist) role. It lets
specialists review user progress reports, assign customized wellness tasks, manage
educational content on mental resilience and self-care, provide professional feedback that
automatically recalculates a user's plan, moderate community groups, and review health
trends to improve the program. It reads and writes health data through the FastAPI
backend, with Supabase Row-Level Security restricting each specialist to their assigned
users.

5.1.4 Admin Subsystem

This subsystem provides administrative control over the platform. It allows admins to view
and manage users, approve member and specialist registrations, suspend and reinstate
memberships, assign roles, monitor user activity, remove inactive programs, and send
announcements and program updates to members.

8.      Deployment Design

8.1. Deployment Diagram

The OneFit system uses a 4-tier cloud-based architecture consisting of the client, frontend,
backend, and database layers. Gym Users, Wellness Specialists, and Administrators access the
system through a web or mobile browser running the Next.js client. The frontend, hosted on Vercel,
handles user interactions and communicates securely with the backend via HTTPS.

The backend, developed using Python and FastAPI, serves as the core of the system. It processes
requests, manages business logic, stores data in a Supabase PostgreSQL database, and
integrates with external services such as USDA FoodData Central for nutrition data, Groq Cloud for
AI-generated recommendations, Hugging Face for pose detection, and an email service for
notifications.

This architecture improves scalability, maintainability, and security by separating responsibilities
across different layers and ensuring that sensitive API credentials remain protected on the
backend.

9.      Summary

This document outlines the software design for the OneFit digital wellness platform, utilising a four-
tier, cloud-based architecture to separate the Gym User, Wellness Specialist, and Admin
subsystems under a shared FastAPI application tier. The design specifies a Client-Server model
with a Next.js frontend hosted on Vercel and a FastAPI backend, supported by a Supabase/
PostgreSQL data tier with Row-Level Security, and integrates external services (Groq Cloud,
Hugging Face, USDA FoodData) for AI plan generation, pose inference, and nutrition data. The
detailed component interfaces, data structures, and deployment plans outlined in this document
demonstrate that the OneFit system is fully specified and ready for the implementation phase. The
design fulfills all functional requirements for a scalable, secure, and user-friendly digital fitness
coaching platform.
